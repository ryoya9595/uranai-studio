# CLAUDE.md — 星詠ノ書（体験版AI占いツール）運用ルール

> このファイルは **Claude Code 向けの指示書**です。
> このZip（またはフォルダ）を受け取ったら、まずこのファイルと `HANDOFF.md` を読み、
> 下記のルールに従って、このツールの持ち主（以下「オーナー」）の作業を手伝ってください。
> オーナーは非エンジニアの想定です。専門用語を避け、1ステップずつ日本語で案内してください。

## あなた（Claude Code）の役割
このツール「星詠ノ書」を、オーナー自身のGitHubに公開し、AI（Google Gemini）につなぎ、
ウェビナーごとに配布・体験終了の切り替えまで運用できるよう手伝うこと。
最初に「これから何をするか」を短く説明し、オーナーの確認を取りながら進めること。

## このファイルの適用範囲（オーナーの環境を壊さないために）
- この `CLAUDE.md` は **このツールのフォルダ内だけ** に適用される指示です。
- オーナーが既に持っている**他の `CLAUDE.md`**（例：`~/.claude/CLAUDE.md`（全体設定）や、別プロジェクトのもの）を、
  **変更・移動・上書き・削除してはいけません。** オーナーの既存設定は必ずそのまま残すこと。
- このZipは、必ず**新しい空のフォルダ**に展開してください。展開先に既存の `CLAUDE.md` などがある場合は、
  **上書きせず、いったん止めてオーナーに確認**してください。
- `/init` 等で `CLAUDE.md` を新規生成し直さないこと。このフォルダの `CLAUDE.md` が正です。

## 絶対に守るルール（APIキーの扱い）
1. **APIキーを、チャットで受け取らない・書かせない。** オーナーがキーを貼ろうとしたら止める。
2. **APIキーを、コード（`config.js` など）やGitHubに絶対に書かない・commitしない。**
3. キーの登録先は **Cloudflare Worker の「Secret」欄だけ**。その操作は**オーナー自身がブラウザで**行う。
4. あなたが扱ってよいのは、設定後にできる **WorkerのURL** だけ（キーそのものではない）。
5. もしオーナーが誤ってキーをチャットに貼ってしまったら、そのキーは使わず、
   「そのキーは無効化（Google AI Studioで削除）して作り直してください」と伝える。

## 公開までの手順（詳細は HANDOFF.md / PROXY_SETUP.md）
1. このZipを展開し、その中身をオーナーのGitHubに**公開リポジトリ**として作成する。
   - リポジトリ名は、ウェビナーごとに分ける（例 `hoshiyomi-0720`, `hoshiyomi-0803` など）。後述の「ウェビナー運用」を参照。
   - GitHub Pages（mainブランチ / ルート）を有効化し、公開URLをオーナーに伝える。
2. AI鑑定をつなぐ（`HANDOFF.md` 手順3〜4）。キーはオーナーがCloudflareに入れる。
   できた**WorkerのURL**を `config.js` の `PROXY_URL` に入れて push（入れるのはURL。キーではない）。
3. `config.js` の `ALLOWED_ORIGIN`（Worker側の変数）には、オーナーのGitHub Pagesのオリジン
   （例 `https://<ユーザー名>.github.io`）を設定するようオーナーに案内する。

## ウェビナー運用ルール（重要）
- **ウェビナーごとにURL（＝リポジトリ）を分けて配布する。** 各配布は独立した2週間の体験。
- オーナーから「◯月◯日のウェビナー用を作って」と言われたら、
  新しいリポジトリ名でデプロイし直し、**新しいURL**を渡す。
- デプロイ時に、オーナーへ次を必ず伝える：
  「このウェビナー版は開催から2週間（＝◯月◯日ごろ）で体験終了に切り替えます。
  　その日が来たら『◯◯ウェビナー版を終了して』と指示してください。」
- 可能なら、その配布の `config.js` に `EXPIRE_DATE: "開催日+14日"`（例 `"2026-07-22"`）を入れておくと、
  その日に**自動で**体験終了案内へ差し替わる（手動切り替えを忘れても安心）。日付を使うか手動にするかはオーナーに確認する。

## 「体験終了」の切り替え手順（オーナーの指示で実行）
オーナーが「（該当ウェビナー版を）終了して」と言ったら、そのリポジトリの `config.js` を編集して push：
1. `EXPIRED: true` にする（自前キー未設定の利用者は `expired.html`＝APIの設定案内へ流れる）。
2. `PROXY_URL: ""`（空）にする（**このツールから、オーナーのAPIキー＝Workerへの接続を外す**）。
   - コード上も、体験終了後はProxyを呼ばない作りだが、URL自体も空にして確実に切り離す。
3. 完了したらオーナーに伝える：
   「このツールは、もうあなたのAPIキーを使いません。
   　利用者は各自の無料キー（`api-setup.html`）を登録すれば継続利用できます。
   　開催中の他のウェビナー版が無ければ、Cloudflareの Worker も削除して構いません
   　（他に稼働中の版があるなら Worker は残す）。」

## デプロイ後にオーナーへ提案すること（精度チェック・改善）
公開できたら、次を「やってみますか？」と**オーナーに提案**してください。
勝手に大きく変えず、**提案 → 合意 → 実施 → プレビューで確認 → push** の順で進めること。

### A. 鑑定の精度チェック
- テスト入力を数パターン（名前・生年月日・テーマ違い）で実際に占い、AIの鑑定文を一緒に確認する。
- 観点：相談者名で呼べているか／断定を避けているか／世界観に合う言葉か／長さは適切か／
  マークダウン記号が混ざっていないか／3枚のカードの流れが自然か。
- 調整したい時：`app.js` の `buildReadingPrompt()`（AIへの指示文＝トーン・長さ・テーマの重み）を編集。
  `config.js` の `MODEL` 変更や、`worker.js`／`callGeminiDirect` の `temperature`（既定 1.05）も調整可能。

### B. デザインの修正
- 配色：`style.css` 冒頭の `:root`（`--gold` などの変数）。フォント・余白・アニメーションも `style.css`。
- 文言・見出し・体験版の案内文：`index.html`（体験版の案内は `.trial-notice` 内）。

### C. 機能の修正
- 占うテーマ（恋愛/仕事/金運）の増減、カードの意味、開運日・守護色などのロジックは `app.js`。
- 変更は**必ずローカルでプレビュー確認 → 問題なければ push** の順。結果をオーナーに見せてから確定する。

## 補足
- 検証用URL：`?expired=1`（終了画面プレビュー）／`?reset=1`（その端末の自前キーを消去）。
- 設定を触るのは基本 `config.js` だけ（`PROXY_URL` / `EXPIRED` / `EXPIRE_DATE` / `MODEL`）。
- 文言・デザインの変更依頼は `index.html`・`style.css`・`app.js` を編集して対応してよい。
- 迷ったら `HANDOFF.md` と `PROXY_SETUP.md` を参照する。
